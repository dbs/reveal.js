The saddest robots.txt a library catalog can offer

User-Agent: *
Disallow: /

AKA "Machines are not welcome here."

While I expect this sort of nonsense from web-hostile proprietary library systems, it was surprising to me to find that even many of the free-as-in-freedom library systems use an absolutely restrictive robots.txt file.

Library catalogs: first principles

Libraries provide multiple access points for resources in catalogues

Users need to know that catalogues exist

Google, Bing, Yahoo are the ready-at-hand search tools

Obligatory Heidegger reference. Library catalogues and other such specialized search tools are, in comparison, present-at-hand: the general user needs to consciously think about using them.

To expand access to our resources ...

We need to expose our access points to third-party search engines

Thus the sadness about absolutely restrictive robots.txt files...

and schema.org is the vocabulary initiated by the search engines

We must not let the perfect be the enemy of the good

If you want to get things done now, and then progressively improve them, then schema.org is a good choice.

Vocabularies are not mutually exclusive: you will be able to publish schema.org from RDA-sourced linked data, for example, whenever (or if) general library systems get around to offering RDA as a source.

Recipe for schema.org catalogue success

  1. Allow machines to crawl your pages: robots.txt and persistent URIs
  2. Embed machine-readable schema.org metadata into ordinary catalogue pages
  3. Generate sitemaps for your record detail pages
  4. Tell the search engines about the sitemaps

Unstructured -> structured -> linked data

  • Bibliographic records
    • Record types: Book, Map, MusicRecording, fallback to CreativeWork
  • Holdings, using the Product/Offer model

Beyond library systems

Bryn Mawr Classical Review

  • Second oldest online scholarly journal in the humanities (1990)
  • Currently implemented in SGML and Tcl (!)
  • Proof of concept: reviews expressed as schema:Review

BMCR: schema:Review and schema:Book

<http://example.org/2011/2011-01-03#review_text> a schema:Review ;
    schema:author "Andreas Markantonatos" ;
    schema:itemReviewed <http://example.org/2011/2011-01-03#review_item> ;
    schema:name "Bryn Mawr Classical Review 2011.01.03" ;
    schema:reviewBody """Ipsem lorum...""" .

<http://example.org/2011/2011-01-03#review_item> a schema:Book ;
    schema:author "Adrian Kelly" ;
    schema:datePublished "2009" ;
    schema:isbn "9780715637135" ;
    schema:name "Sophocles: Oedipus at Colonus." ;
    schema:numberOfPages "176" ;
    schema:publisher "Duckworth" .

Addressing gaps in schema.org

In 2013 schema.org moved to W3C WebSchemas as the public forum for enhancement proposals

Enter the W3C schema.org Bibliographic Extension Community Group

  • AKA SchemaBibEx
  • A rag-tag group of linked data bibliographic enthusiasts trying to establish consensus
  • Using open source library systems (Evergreen, Koha, VuFind) as proofs of concept / reference implementations
      • It was nice to participate in a group in which the discussions were not purely theoretical, but grounded in practice (even if I was the one doing most of the grounding at the time).

Some schemabibex wins

  • Early: moving "citation" from MedicalScholarlyArticle to CreativeWork
  • Middle: Holdings-as-Offers as a best practice, with decommercialized descriptions
  • Most recently: Periodicals and CreativeWork relationships
