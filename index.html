Wikibase

  • Open source (GPL v2 or later) project on which Wikidata is built
  • It scales!
    • 50M items
    • 5,000 properties
    • 450M claims

Why Wikibase?

  • It exists now
  • Complete package for collaboratively building and maintaining a dataset
  • Active development community
    • Extensions and development efforts might be embraced
    • Skills learned may be reused in other projects
  • Good platform for rapid prototyping
  • In my first half day, I had modelled the ORKG ontology and was ready to begin adding research data

Core features built-in

  • Rich data model
  • Basic item and property editor interface
  • Per-page discussion and revision history
  • Authentication and authorization controls
  • Multilingual
  • Self-hosting ontology
  • Content negotiation for per-item or bulk RDF dumps
  • MediaWiki API with bindings in Python, Java, JavaScript, etc
  • Integration with OpenRefine 3.0

SPARQL endpoint

  • Built on Blazegraph
  • Autocomplete query editor
  • Various visualizations (maps, graphs, etc)
  • Loosely coupled via RDF updates

11 built-in datatypes

  • Media file (maintained in Wikimedia Commons)
  • External identifier
  • Geographic coordinates
  • Geographic shape
  • Item
  • Monolingual text
  • Point in time
  • Property
  • Quantity
  • String
  • Tabular data (maintained in Wikimedia Commons)
  • URL

Community

  • Channels on IRC, mailing lists, Wikidata itself, Phabricator (project management software), Discourse
    • Initial response was slow, but Wikimania prep + unknown newbie == limited cycles
    • Weekly Technical Advice IRC Meeting was perfect
  • Docker images via docker-compose (8 images in all)
    • Images were a release behind, so I updated them

Maybe missing

  • RDF for every revision
  • Broad RDFS or OWL ontology support
    • Much of this could be added relatively easily
  • Broad adoption by independent projects
    • This may be an opportunity for ORKG

ORKG observations

  • Current focus is on the technology stack
  • There is a huge amount of work to do to design, build, test, and document a full technology stack
  • ORKG is missing the explicit use cases and prototypes to guide that work, however:
    • Granularity of statements to be added to the graph
    • Just URIs for publications, or more?
      • If more, reuse an existing publication ontology or mint a new one?
    • CITO for roughly 40 citation types instead of 11 relations in ORKG?
  • Data entry techniques require different prototypes:
    • Form linked from conference / article submission
    • Retrospective conversion
    • Automated vs manual approaches
    • Creation vs. editing vs. administration
  • Analysis goals
    • Generated systematic reviews?
    • Research area summaries and trends?
